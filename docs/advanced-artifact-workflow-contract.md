# Advanced Artifact Workflow Contract

Phase E keeps ICA review, bad-channel interpolation review, and manual artifact
annotation review out of the execution path. This contract defines the JSON
shape and product boundaries for those future actions so export, reports, and
rerun planning can reason about them before one-click execution exists.

Schema:

- `docs/schemas/artifact-action.schema.json`

Fixture:

- `tests/fixtures/artifacts/artifact_action_manual_review_v1.json`

## Scope

In scope for this contract:

- Artifact actions that record proposed or applied changes to bad channels, ICA
  components, artifact annotations, or review notes.
- Pending/manual-review states that can be shown in QC and exported for audit.
- Report and export rules for pending actions.

Out of scope for this slice:

- Applying an artifact action to data.
- Creating derived preprocessing, epoch, ERP, comparison, report, or export runs.
- Interactive waveform/topography review UI.
- Changing the current preprocessing worker behavior.

## Artifact Action Schema

An artifact action is a small auditable JSON document. It may be stored as a
diagnostic artifact in a run directory and referenced from a future action list.
It is not a replacement for `PreprocessingConfig`; it records a proposed or
reviewed change that may later become a config patch for a new run.

Required top-level fields:

- `schema_version`: currently `1`.
- `action_id`: stable action id.
- `action_kind`: one of `mark_bad_channels`, `interpolate_bad_channels`,
  `ica_exclude_components`, `reject_artifact_annotations`, or
  `manual_review_note`.
- `status`: lifecycle state.
- `review_state`: whether manual review is required and whether it is pending,
  approved, or rejected.
- `source`: dataset, run, manifest, and QC context where the action was proposed.
- `targets`: channels, ICA components, annotations, and time windows affected by
  the action.
- `config_patch`: future rerun config patch derived from the action.
- `effects`: whether the action mutates signal, requires a new run, creates
  artifacts, or invalidates downstream outputs.
- `export_report_policy`: report/export inclusion behavior.
- `diagnostics`: structured warnings using the existing warning taxonomy.

## States

The action lifecycle uses explicit states:

- `proposed`: generated by QC or user selection, not ready to apply.
- `pending_manual_review`: requires human review before use in a derived run.
- `approved`: reviewed and eligible to become a rerun config patch.
- `applied`: already used by a completed derived run.
- `rejected`: reviewed and rejected.
- `skipped`: intentionally not applied.
- `failed`: attempted but could not be applied.
- `superseded`: replaced by a newer action.

`review_state.state` is narrower:

- `not_required`
- `pending`
- `approved`
- `rejected`

The invariant is:

- `status=pending_manual_review` must have `review_state.required=true` and
  `review_state.state=pending`.
- Pending review does not make the source run incomplete.
- Pending review does not automatically invalidate existing artifacts; it only
  marks future interpretation and rerun planning as review-sensitive.

## Action Kinds

### `mark_bad_channels`

Records channels to mark as bad in a future preprocessing config. It does not
edit the current run.

Expected targets:

- `targets.channels`: one or more channel names.

Expected effects:

- `mutates_signal=false` if only marking bads for review.
- `requires_new_run=true` if the action should affect downstream preprocessing.

### `interpolate_bad_channels`

Records channel interpolation intent. This action changes the signal only when a
new preprocessing run applies the resulting config.

Expected targets:

- `targets.channels`: one or more channel names.

Expected effects:

- `mutates_signal=true`
- `requires_new_run=true`
- `invalidates_downstream` includes `preprocessing`, `epoch`, `erp`,
  `comparison`, `report`, and `export`.

### `ica_exclude_components`

Records ICA components selected for exclusion after component review.

Expected targets:

- `targets.ica_components`: zero-based component indexes.

Expected effects:

- `mutates_signal=true`
- `requires_new_run=true`
- `invalidates_downstream` includes `preprocessing`, `epoch`, `erp`,
  `comparison`, `report`, and `export`.

### `reject_artifact_annotations`

Records manual artifact time windows or annotation ids to reject in a future
run.

Expected targets:

- `targets.annotations`
- `targets.time_windows`

Expected effects:

- `mutates_signal=true` when the action removes or rejects signal segments.
- `requires_new_run=true`.

### `manual_review_note`

Records a review note without changing configuration.

Expected effects:

- `mutates_signal=false`
- `requires_new_run=false`

## Export And Report Impact

Pending actions are audit information, not fatal export errors.

`analysis_report.json` should include an `artifact_actions` section when action
artifacts exist. Each item should preserve `action_id`, `action_kind`, `status`,
`review_state`, `targets`, `effects`, and diagnostics. Pending manual review
should surface as a warning in the report diagnostics.

`export_bundle_manifest.json` should include an `artifact_actions` section when
action artifacts exist. The source action JSON should be bundled under
`diagnostics/artifact_actions/`. Pending manual review should not block ZIP
creation because the source run remains completed and reproducible.

Pending manual review should not block ZIP creation.

`artifact_manifest.json` should list action documents as diagnostic artifacts:

```json
{
  "logical_name": "artifact_action_artifact-action-001",
  "artifact_type": "artifact_action_json",
  "path": "artifact_action_artifact-action-001.json"
}
```

The report/export rule is:

- Include pending actions.
- Warn on pending manual review.
- Do not silently apply pending actions.
- Do not mutate or delete existing run artifacts.
- Derived reruns must reference the approved action id in their provenance.

## Rerun Planning Impact

Future rerun planning should inspect approved and pending artifact actions:

- Pending manual-review actions produce warnings and keep `can_rerun=true` only
  when the rerun does not depend on the pending action.
- Approved actions can be converted into config patches for a new run.
- Applied actions should point to the derived run that consumed the action.
- Rejected, skipped, and superseded actions remain reportable audit records but
  should not be applied.

This keeps one-click rerun deterministic: a rerun can only apply actions whose
review state is final and whose config patch is explicit.
